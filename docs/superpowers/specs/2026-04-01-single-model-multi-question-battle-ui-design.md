# 单模型多问题大乱斗 UI 重构设计

## 背景

当前 `单模型多问题大乱斗` 页面将历史记录、运行配置、题目编辑、执行控制、结果轨迹堆叠在同一条线性页面中。功能虽然完整，但首屏注意力被历史记录抢占，配置区和结果区缺少主次关系，首次使用者难以快速判断应该先做什么，重度使用者也需要在长页面中来回滚动。

本次重构不改变后端 API、评测模式、历史复用、分享、轨迹节点详情等核心能力，重点重构前端的信息架构、视觉层级和交互组织方式，让页面同时满足两类目标：

- 新用户打开页面后可以快速理解主流程并完成第一次评测。
- 高频评测用户能够在同一页面内完成配题、启动、观察运行和复盘结果。

## 目标

- 将页面重构为“实验台式指挥舱”。
- 以渐进式信息架构组织复杂功能，先暴露主流程，再按需展开高级控制。
- 保留当前已有能力：历史加载、复用为新任务、分享、取消执行、轨迹点击查看详情。
- 明显降低首次使用的理解成本。
- 保持桌面端高效同屏工作流，并兼顾移动端可用性。

## 非目标

- 不新增新的评测能力或后端接口。
- 不引入题库模板、批量导入、批量导出等新产品能力。
- 不改变 `QuestionTrajectoryGraph` 和 `DetailDrawer` 所表达的核心语义，只重构其外层容器、层级和视觉表现。

## 已确认的设计决策

- 视觉方向：`A. 指挥舱实验台`
- 页面结构：`A. 渐进式实验台`
- 用户定位：兼顾新手与重度用户，但偏重度用户
- 历史处理：首页仅展示最近几条，完整历史通过展开或二级视图进入
- 改动边界：允许 UI 和交互重组，但保留现有核心能力和 API
- 成功标准优先级：首次上手更快

## 体验原则

### 1. 先让用户知道下一步做什么

页面首屏必须明确回答三个问题：

- 这里是做什么的
- 当前是否已经准备好可以开跑
- 下一步应该点哪里

因此首屏必须先呈现页面标题、简要说明、主操作按钮和核心状态卡，而不是历史记录或完整表单。

### 2. 让复杂功能保持“可见但不打扰”

裁判阈值、并发上限、历史记录、结果细节、节点详情都属于高价值但非第一步信息。这些内容需要保留，但默认只展示必要摘要，通过折叠、抽屉、次级区块逐层展开。

### 3. 配置与结果同屏

该页面本质上不是一次性表单，而是实验台。用户需要边改配置边观察运行状态和结果，因此桌面端不采用分步骤切页，而采用左侧主工作区加右侧观测区的双栏结构。

### 4. 历史为辅助，而不是入口门槛

历史记录是高频辅助能力，但不应在首次进入时抢占主视线。首页仅展示最近几条记录和关键操作，完整历史通过显式展开查看。

## 信息架构

页面从上到下分为四层：

1. 顶部指挥头部
2. 核心概览卡
3. 双栏实验台主体
4. 最近历史摘要

### 1. 顶部指挥头部

头部包含：

- 返回模式选择入口
- 页面标题 `单模型多问题大乱斗`
- 一句价值说明：强调“选模型、配题、开跑、看稳定性”
- 主操作区：`开始评测`、`取消`、`新任务`、`分享`
- 当前运行状态 Badge

该区域负责建立页面身份和操作锚点。无论页面滚动到何处，用户都应清楚自己在哪里、当前处于什么状态、最关键的动作是什么。

### 2. 核心概览卡

头部下方使用四张高密度状态卡，统一展示：

- 参赛模型 / 裁判模型是否已选
- 当前题目规模与总执行次数
- 运行进度与运行状态
- 稳定性结果摘要

这些卡片承担“系统状态仪表盘”的作用。未开始时提供准备度反馈，运行中提供进度感，完成后提供总结入口。

### 3. 双栏实验台主体

桌面端采用 `左主工作区 + 右观测区`。

左侧主工作区分两个主分区：

- `Step 1. 基础配置`
- `Step 2. 题目工作区`

右侧观测区分两个主分区：

- `运行观测台`
- `问题轨迹与结果`

左侧负责输入，右侧负责反馈，避免用户频繁上下滚动找结果。

### 4. 最近历史摘要

历史区域从页面顶部移到底部，只露出最近 3 条记录。每条记录保留：

- 任务标题 / 编号
- 创建时间
- 状态
- `查看记录`
- `复用为新任务`

区域底部提供 `展开全部历史` 入口。这样既保留高频历史能力，又不破坏首次使用的主流程。

## 关键区域设计

### 顶部指挥头部

视觉上采用深色高对比“舱室条”样式，形成页面记忆点。主按钮 `开始评测` 使用高优先级强调色，其他操作保持次级按钮样式。运行中时，头部状态与概览卡状态必须联动，避免同一页面出现多个互相矛盾的状态表达。

### Step 1. 基础配置

该区只保留最影响开跑的配置：

- 参赛模型
- 裁判模型
- 裁判阈值
- 并发上限

其中模型选择始终可见；裁判阈值和并发上限属于高级参数，视觉上弱于模型选择，但仍放在同一区，避免用户到别处寻找。若后续需要进一步降低认知负担，可将阈值与并发做成 `高级参数` 折叠区，但首版仍保持可见，防止配置被隐藏得过深。

### Step 2. 题目工作区

题目区不再只是简单重复表单，而是重构为“题目卡片列表”。每张题目卡片包含：

- 标题行：题号、题目名称摘要、runs/passK 摘要、删除操作
- 元信息行：Question ID、标题
- 内容编辑区：题目、期望答案
- 执行规则行：runs、passK

每张卡片需要有更清晰的内部分组和行列节奏，让用户快速扫描哪些题目配置完整，哪些题目仍为空。

### 运行观测台

右侧上半区用于回答“现在跑到哪里了”：

- 当前任务状态
- 已完成 / 总任务数
- 当前进行中的题目或尝试
- 失败数 / 待执行数 / 成功数
- 必要时显示全局错误摘要

该区域需要比当前实现更像状态面板，而不是结果区前的过渡空白。

### 问题轨迹与结果

右侧下半区承载 `QuestionTrajectoryGraph`，但外层容器要升级：

- 增加结果说明文案，告诉用户“点击节点查看详情”
- 在标题区直接显示稳定性摘要与通过题数
- 每个问题卡片在视觉上强化 `题目名称 / passK / 通过数`
- 节点颜色与头部状态卡、运行观测台颜色体系统一

点击节点后仍使用现有 `DetailDrawer` 模式查看详情，避免为本次重构引入新的结果详情范式。

## 关键交互设计

### 首次进入

首次进入页面时，用户先看到：

- 页面用途说明
- 还未准备完成的状态提示
- 模型选择和题目区的第一张空题目卡
- 右侧结果区空状态说明

空状态文案必须明确表达：

- 先选择参赛模型和裁判模型
- 至少填写一道题目与期望答案
- 点击开始评测后，右侧将出现运行进度与结果

### 开始评测

点击 `开始评测` 后：

- 头部状态切换为运行中
- 概览卡显示总题数、总尝试数与当前进度
- 左侧表单进入只读或半只读状态
- 右侧观测区立即进入活跃状态

用户不需要滚动到页面底部寻找结果，运行反馈必须在当前视线中出现。

### 取消 / 新任务 / 分享

- `取消` 仅在运行中可用，并与当前 runId 绑定
- `新任务` 清空执行结果，但保留当前配置作为新的起点
- `分享` 在已有运行结果后可用，并在头部或结果区附近回显分享链接

分享链接反馈不再作为突兀插入卡片，而应融入当前指挥头部或结果摘要区的反馈流中。

### 查看历史与复用

历史区仍支持两条主链路：

- `查看记录`：加载并查看该次 run 的完整结果
- `复用为新任务`：把模型与题目配置加载到当前工作台，但执行状态清空

当页面由历史配置填充时，需要在配置区顶部显示轻量提示条，告知用户当前正在基于哪条历史记录继续工作。

### 节点详情抽屉

`DetailDrawer` 继续保留抽屉式细节查看方式。重构范围仅包括：

- 打开入口的可发现性更强
- 抽屉标题与节点上下文更明确
- 与外部状态体系保持一致的颜色和标签

不改变抽屉中的输出、推理、裁判结果、工具调用等信息结构。

## 视觉系统

### 总体风格

采用高对比、深色技术感的“实验台指挥舱”风格。整体以暗色顶部和暗色右侧观测区建立专业评测氛围，左侧配置区保持浅色高可读工作面，形成“输入区明亮、观测区沉浸”的对照关系。

### 色彩

- 主背景延续当前系统主题变量，不引入孤立硬编码色板作为全局基础
- 深色舱面强调色参考 `#020617`, `#0F172A`, `#1E293B`
- 正向状态使用绿色系，运行中使用主品牌色，错误使用 destructive 色，评测中使用 amber 系
- 所有状态颜色同时通过文案和图标表达，不依赖颜色单独传达意义

### 排版

- 保持当前产品字体体系，避免只为该页单独替换全局字体
- 页面标题、步骤标题、状态卡数值、题目卡摘要形成清晰层级
- 辅助说明文案统一收敛为小号 muted 文本，避免页面噪声

### 图标与动效

- 继续使用 Lucide 图标集
- 动效保持克制，仅用于运行中、节点状态和按钮反馈
- 所有动画应尊重 `prefers-reduced-motion`

## 响应式策略

### 桌面端

- 双栏布局固定存在
- 右侧观测区可使用 sticky 行为，保证运行状态和结果始终在可视范围
- 题目卡片支持更丰富的双列编辑布局

### 平板端

- 维持纵向主流程，但可保留概览卡的双列排布
- 观测区下移到配置区之后

### 移动端

- 所有区域改为单列堆叠
- 先显示头部和主按钮，再显示概览卡、基础配置、题目工作区
- 运行观测与最近历史默认压缩显示，避免首屏过长

## 可访问性与可用性

- 所有按钮、可点击节点、历史记录操作都必须具备明确 hover 与 focus-visible 状态
- 题目卡片中每个输入框维持清晰的标签，不以 placeholder 代替 label
- 运行状态、成功、失败、评测中都必须有文字状态
- 移动端不能出现水平滚动
- 折叠区域打开和关闭的状态要有可感知反馈

## 组件拆分建议

当前 `SingleModelMultiQuestionPageClient.tsx` 体积过大。本次重构应顺带完成目标导向的拆分，但拆分范围只服务于新 UI 组织，不做无关抽象。

建议拆分为：

- `SingleModelBattleHero`：页面头部与主操作
- `SingleModelBattleOverview`：四张概览卡
- `SingleModelBattleConfigPanel`：基础配置区
- `SingleModelQuestionWorkspace`：题目卡片列表与单题卡片
- `SingleModelBattleMonitorPanel`：运行观测台
- `SingleModelBattleResultsPanel`：结果摘要与轨迹容器
- `SingleModelBattleHistoryPanel`：最近历史摘要

主页面客户端组件仍保留状态编排和数据流控制，子组件负责展示和局部交互。

## 测试策略

本次重构必须补足以下前端测试：

- 页面首屏默认展示顺序与关键文案
- 历史记录默认只展示最近若干条并提供展开入口
- 运行状态切换后头部与概览卡同步更新
- `复用为新任务` 仍能正确回填配置并清空执行态
- 结果区空状态、运行态、完成态的关键文案正确
- 关键主按钮在不同状态下的 disabled 逻辑保持正确

测试以 `Vitest + Testing Library` 为主，不依赖视觉快照。

## 实施约束

- 不改变 battle 相关 API 契约
- 不修改 share、history、detail 的业务含义
- 不引入新的状态管理库
- 尽量复用当前 `DetailDrawer`、`QuestionTrajectoryGraph`、`ModelSelector` 等已有能力

## 预期结果

重构完成后，用户在进入页面的 5 秒内应能够理解：

- 这是一个给单个模型做多题稳定性评测的页面
- 先选模型，再配题，再开始评测
- 右侧会实时展示运行过程与结果
- 历史记录仍在，但不会挡住首次开跑的主路径

页面整体观感应从“长表单 + 结果卡片堆叠”升级为“可直接操作的评测实验台”。
